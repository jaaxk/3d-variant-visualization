# af2_modeling

Structure-prediction workflow for the full PKD1/PKD2 complex, using this
cluster's existing official AlphaFold2 2.3.2 install
(`/share/apps/af2/2.3.2-20231225/`) rather than building ColabFold from
scratch. Separate from `protein_vis` (which only visualizes already-existing
structures) -- this produces the structure file that later gets fed into
`protein_vis` via its existing `file:<path>` structure spec.

## Why the full complex, and why this AF2 install

Prior structure-prediction attempts used the hosted AlphaFold Server, which
caps jobs at 5,000 total residues -- that's a hosted-service resource limit,
not a property of AlphaFold2/ColabFold themselves. On this cluster the only
real constraint is GPU memory/time, so this attempts the full, untrimmed
complex: 1x PKD1 (UniProt P98161, 4303 aa) + 3x PKD2 (UniProt Q13563, 968 aa
each, matching the known 1:3 PKD1:PKD2 stoichiometry) = **7,207 residues** --
well beyond any published AlphaFold2/AlphaFold-Multimer benchmark (typically
~2-3k residues), so this is a genuine feasibility experiment.

The cluster already has a fully-provisioned official AlphaFold2 install with
every database pre-mounted (uniref90, mgnify, bfd, uniref30, pdb70,
pdb_mmcif, params -- all under `/projects/work/public/ml-datasets/af2/`), so
no new environment/overlay was needed. The exact invocation chain (confirmed
by reading `/share/apps/af2/2.3.2-20231225/run-alphafold.py` directly): you
call that script with standard AlphaFold CLI flags
(`/share/apps/af2/pyenv/bin/python3 run-alphafold.py --fasta_paths=...`); it
internally shells into a Singularity wrapper that mounts all the databases
and runs the container's own bundled `run_alphafold.py` with `--nv` GPU
passthrough (auto-detected). We never touch Singularity/overlays ourselves.
`TF_FORCE_UNIFIED_MEMORY=1` / `XLA_PYTHON_CLIENT_MEM_FRACTION=4.0` (letting
JAX address beyond the GPU's raw VRAM) are already set inside that script.

## Why grafting instead of relying on AlphaFold's template feature

AlphaFold's template mechanism feeds a template's structure into the network
as a learned input feature -- it never copies template coordinates directly
(that's classical homology modeling, which AlphaFold doesn't do). So even
with 6A70 as a template, the output in that region is influenced by, but not
identical to, 6A70's real deposited coordinates. To guarantee the
6A70-covered region is byte-identical to the experimental EM structure, we
superpose the AlphaFold prediction onto 6A70 (using the residues both share)
and splice: keep 6A70's real atoms for the shared region, keep AlphaFold's
atoms only for the genuinely new region it predicted. This is independent of
whichever templates AlphaFold's own automatic search happens to pick.

## Files

- `fasta/pkd1_pkd2_complex_full.fasta` -- 1x full-length PKD1 + 3x PKD2,
  generated by `scripts/build_complex_fasta.py`.
- `scripts/build_complex_fasta.py` -- assembles the multi-chain FASTA.
  `--pkd1-start N` trims PKD1's N-terminus (for the fallback run below);
  default is full-length/untrimmed.
- `scripts/watch_af2_progress.py` -- runs the AF2 command as a subprocess,
  shows a live elapsed-time heartbeat plus timestamped stage-change lines
  (parsed from AF2's own log output) via tqdm -- AF2's CLI has no built-in
  progress indicator, and at this scale whether it's converging in a
  reasonable time is genuinely unknown up front.
- `scripts/graft_6a70_onto_prediction.py` -- the superpose+splice step above.
- `slurm/run_af2_predict.sh` -- generic, parameterized job script (one H200
  GPU, 500G mem, 16 cpus, 48h ceiling -- the real "is this reasonable" call
  is made by watching the live progress output and `scancel`-ing if it's
  clearly not converging, not by the SLURM time limit itself).
- `slurm/submit_pkd1_pkd2_full.sh` -- thin wrapper for the full 7,207-residue
  attempt (`num_predictions_per_model=1`, for a faster feasibility read than
  the default of 5).

## If the full complex doesn't finish in a reasonable time

Rebuild the FASTA with PKD1 trimmed but with generous overlap into 6A70's
resolved region (~residues 2221-4303):

```bash
python scripts/build_complex_fasta.py --pkd1-start 1900 \
    --output fasta/pkd1_pkd2_complex_partial.fasta
sbatch slurm/run_af2_predict.sh \
    fasta/pkd1_pkd2_complex_partial.fasta \
    /scratch/jv2807/pkd1/af2_predictions/pkd1_pkd2_complex_partial \
    pkd1_pkd2_complex_partial 1 best
```

## Possible future optimization (not built here)

The cluster's AF2 install also has a genuine CPU-only "MSA-only" mode
(confirmed by reading `run_alphafold-msa-only.py` -- its `predict_structure()`
only runs feature/MSA generation and pickles `features.pkl`, with the entire
JAX/GPU model-inference block commented out), which could split the
CPU-bound MSA search from GPU inference so the GPU isn't allocated (and
idle) during MSA generation. The NYU wrapper ties that mode to a smaller
"reduced-dbs" database preset rather than the full databases, so using it
with full-quality MSAs would need custom plumbing -- out of scope unless
iterating on this becomes routine enough to be worth the extra effort.
